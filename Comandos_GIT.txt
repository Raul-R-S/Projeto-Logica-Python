CONEXÃO COM O GITHUB (SÓ FAZ UMA VEZ POR PROJETO)
=========================================

1. git remote add origin [link]
   - O que faz: Diz ao PC onde é a casa do projeto na nuvem.
   
2. git branch -M main
   - O que faz: Define "main" como o nome do tronco principal.

3. git push -u origin main
   - O que faz: Envia os arquivos e "conecta" as duas pontas para sempre.

1. git status
   - O que faz: Mostra o estado atual (quem mudou, quem é novo).
   - Quando usar: SEMPRE. Antes, durante e depois de qualquer comando.

2. git add [nome_do_arquivo] ou git add .
   - O que faz: Prepara os arquivos para o salvamento.

   - O que faz: Tira a "foto" oficial e registra no histórico.
   - Cenário: Use quando chegar em um ponto do código que está funcionando.

4. git push
   - O que faz: Envia suas fotos (commits) do PC para o GitHub.
   - Cenário: Use no final do dia ou quando quiser que o RH veja seu progresso.

PARTE 2: CONECTANDO O PC AO GITHUB (O APERTO DE MÃO) 
=====================================================

1. git remote add origin [LINK_DO_REPOSITORIO]
   - Cenário: Quando a pasta no PC já existe mas o GitHub ainda não a conhece.
   - O que faz: Cria a ponte entre o seu Google Drive e a nuvem.

2. git branch -M main
   - Cenário: Ajuste de padrão. 
   - O que faz: Garante que o nome do projeto seja "main" (padrão profissional).

3. git push -u origin main
   - Cenário: O grande envio!
   - O que faz: Sobe os arquivos. O "-u" serve para o PC "lembrar" que, 
     daqui para frente, basta digitar apenas "git push" para atualizar.

------- O "Segurança da Festa": .gitignore

Você perguntou como dizer ao Git o que não salvar (como senhas da Orion ou pastas pesadas do Python). 
Para isso, usamos um arquivo especial chamado .gitignore (o ponto no início do nome é obrigatório).

Como criar:
    Crie um novo arquivo no VS Code chamado .gitignore.
    Dentro dele, escreva o nome do que quer ignorar. Exemplo:

# Ignorar arquivos de senhas
    senhas.txt
    config_banco.json

# Ignorar pastas que o Python cria sozinho
    __pycache__/
    O Resultado: 
    O Git vai passar por esses arquivos e fingir que eles não existem. Eles ficam no seu Google Drive, 
    mas nunca sobem para o GitHub.

-------Comando vs. Ícones do VS Code (O que é melhor?)

o VS Code tem ícones (na barra lateral esquerda, um ícone que parece um triângulo com círculos).
Pelo Ícone (Visual): É bom para ver rápido o que mudou e clicar no "+" para adicionar. É mais "amigável".
Pelo Terminal (Comandos): É o jeito Expert.
Vantagem: 
Se um dia o VS Code der erro, você sabe resolver. Se você precisar mexer no servidor da empresa (onde não tem tela, só texto), você saberá o que fazer.
Minha sugestão: 
Como estamos na mentoria, continue pelos comandos. Isso vai te dar uma autoridade técnica muito maior quando estiver conversando com outros programadores.

📂 PARTE 1: Criando um novo projeto e sincronizando no GitHub
Use esta etapa apenas uma vez por projeto, quando estiver começando do zero.

1ª Etapa - Preparação no GitHub
Vá ao site do GitHub e crie um novo repositório (como fizemos com o Raul-R-S). 
Não marque a opção de adicionar README se você já tiver arquivos na sua pasta do PC. Copie o link https://... que ele vai te dar.

2ª Etapa - Inicialização Local (Terminal)
No VS Code, abra a pasta do projeto e digite:

git init

Por que fazer: Esse comando cria a pasta invisível .git. 
Sem isso, o seu computador não sabe que aquela pasta deve ser vigiada pelo Git. É o "Certificado de Nascimento" do projeto.

3ª Etapa - O Primeiro Carimbo
git add .
git commit -m "Início do projeto: Estrutura inicial"

Por que fazer: O add . prepara todos os arquivos e o commit tira a foto oficial. 
Você precisa dessa "foto" inicial para ter o que enviar para a nuvem.

4ª Etapa - Criando a Rota e o Nome
git branch -M main
git remote add origin https://SEU_TOKEN@github.com/Raul-R-S/NOME-DO-PROJETO.git

Por que fazer: O primeiro comando garante que o nome da sua ramificação seja main (padrão de mercado). 
O segundo cria a "ponte" direta para o GitHub usando o seu Token para não pedir senha.

5ª Etapa - O Primeiro Empurrão
git push -u origin main

Por que fazer: O -u diz ao Git: "Lembre desse caminho". Nas próximas vezes, você não precisará escrever o endereço todo de novo.




🔄 PARTE 2: Dia a Dia (Evoluindo e Salvando)
Use esta etapa sempre que você alterar um código, criar uma View SQL ou melhorar o main.py.

1ª Etapa - Verificar o que mudou
git status

Por que fazer: Antes de salvar, você precisa saber o que o Git está vendo. 
Ele vai te mostrar em vermelho o que foi alterado. É o seu "check-list" de segurança.

2ª Etapa - Preparar para o envio
git add .

Por que fazer: Você está dizendo: "Git, pegue todas as melhorias que eu fiz e coloque na fila para o salvamento".

3ª Etapa - Registrar a Versão (A Mensagem é a Chave)
git commit -m "Melhoria na lógica de cálculo de impostos no main.py"

Por que fazer: Como você busca senioridade, suas mensagens devem ser claras. Se amanhã o código der erro, 
você saberá exatamente em qual "foto" (commit) a mudança aconteceu.

4ª Etapa - Sincronizar com a Nuvem
git push

Por que fazer: Este comando pega suas alterações locais e joga para o GitHub. 
É aqui que seu portfólio se atualiza e seu backup fica garantido.

💡 Dica de Ouro do "Cofre" (Segurança)
Sempre que o projeto envolver dados da empresa ou senhas, lembre-se do arquivo .gitignore.

Crie um arquivo chamado .gitignore na raiz da pasta.

Escreva dentro dele o nome dos arquivos que não podem ir para a nuvem (ex: senhas.txt ou *.key).
